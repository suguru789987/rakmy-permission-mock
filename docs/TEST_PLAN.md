# 検証プラン（エンジニア実行 ＋ PdM受入）

権限設定 本番実装の**実行可能な検証計画**。エンジニアは「実行手順・ケース・カバレッジ」を、PdMは「受入基準・エビデンス・サインオフ」を見る。
- 合格点の枠組み＝`VERIFICATION.md`、具体ケース＝`TEST_CASES.md`、判定キー＝`feature_keys.json`、設計＝`ENFORCEMENT.md`。
- 本書はそれらを**トレーサビリティ（受入条件↔ケース）と網羅（全feature_key×ロール）とサインオフ**で束ねる。

---

## 0. 読み方
- **エンジニア**：§1 環境/フィクスチャ → §3 カバレッジ → §4 実行手順 → §5 ケース。CIで自動化。
- **PdM**：§6 受入基準・エビデンス・サインオフ。各フェーズの「提示物」を見て定量で合否判定。

## 1. 環境・前提・フィクスチャ
- **環境**：本番同等のステージング＋フィーチャーフラグ（`permissions.shadow` / `permissions.enforce.<phase>`）。本番本体は非破壊。
- **シードスクリプト**：`db/seeds/permission_test.rb`（下記フィクスチャを冪等投入）。**全テストはこのシード前提**。
- **フィクスチャ（`TEST_CASES.md` §1 を正とし、機能型を網羅）**：
  - 店舗 S1/S2/S3、ユーザー6（店長@S1・従業員@S1・エリア長[S1,S2]・経理(all)・オーナー・兼務）。
  - ロール権限：代表だけでなく**機能型を1つ以上含む**：
    - edit＋削除あり（vendor）／edit＋削除なし／**inheritFrom連動**（budget.monthly_wizard＋month_cost）／**noStaff**（emp_view・uicustom.dnd_shop）／**action**（data.download・dnd）／**admin**（company・link）／**fixed会社専用**（employees.payroll・closing）。
  - 指標：open/store/company × allowlist 有無（TEST_CASES §1）。
- **店舗スコープ対象モデル一覧**（越境を全モデルで検証する母集合）：Vendor / Delivery / Sales / Purchase / Employee / Inventory / PettyCash / Budget / Cost …（`for_user` を当てる全モデル）。

## 2. 検証の型（どう確かめるか）
| 型 | ツール/方法 | 適用 |
|---|---|---|
| 機能判定 | RSpec **request spec**（ロール別に画面/API叩く） | P0-P2,P4 |
| シャドー差分 | ログ集計SQL（feature_key×ロール×過剰ブロック件数の日次） | P0 |
| 越境（スコープ） | 他店ID直指定・横断クエリの**ネガティブテスト**＋件数比較 | P3 |
| 性能 | k6/JMeter で**ベースライン取得→負荷→p95比較**・bullet（N+1検出） | P3 |
| 監査/移行 | audit_logs アサート・移行 before/after の判定 diff スクリプト | P5 |
| 網羅 | `feature_keys.json` から**全キー×適用ロールのケースを生成**し未テスト0を担保 | 全 |

## 3. カバレッジ方針（網羅の担保）
- **原則：全 feature_key × その機能が適用される全ロール を最低1ケース**。`feature_keys.json` を読み、`(feature_key, role)` の組を生成→各組に「閲覧/操作/削除/越境」の該当ケースが存在することを CI で検査（未カバー=fail）。
- **必須の境界ケース**（取りこぼしやすい）：
  1. **削除境界**：操作(登録+編集) と 全て(+削除) の差（TC-P2-05/06）。
  2. **案C連動**：登録↔編集連動・操作なし→削除off（TC-P2-03/04）。
  3. **noStaff**：従業員は emp_view/dnd_shop を取得不可。
  4. **fixed会社専用**：店舗ロールで employees.payroll/closing=403。
  5. **inheritFrom**：親(月次予算設定)の権限に子(単月費用設定)が追従。
  6. **全モデル越境**：§1 の全店舗スコープモデルで他店データ=0件。
  7. **和集合**：兼務(店長＋経理)は all 優先。
- **カバレッジ報告**：CI が「テスト済 (feature_key,role) / 全組」を出力。**100%でP受入の前提**。

## 4. 実行手順（エンジニア）
```bash
# 1) シード
bin/rails db:seed:permission_test
# 2) 機能判定テスト（フェーズ別タグ）
bundle exec rspec spec/permissions --tag p1   # P1だけ等
# 3) カバレッジ検査（未カバーの (feature_key,role) があれば fail）
bundle exec rake permissions:coverage_check
# 4) シャドー集計（P0・直近N日）
bundle exec rake permissions:shadow_report DAYS=14   # 過剰ブロック件数を画面別出力
# 5) 性能（P3）
k6 run perf/scope_baseline.js && k6 run perf/scope_loaded.js  # p95比較
# 6) 移行diff（P5）
bundle exec rake permissions:migration_diff   # master/manager/shop_manager→新ロールの判定一致を出力
```
- RSpec の書き方は `ENFORCEMENT.md §11` の `authorize_feature!`/`for_user` を叩く request spec。各ケースは **Given(フィクスチャ)→When(操作)→Then(期待)**。

## 5. ケースライブラリ
- 具体ケース（前提データ＋入力＋期待結果）は **`TEST_CASES.md`**（P1〜P5・約28＋境界）。本計画はこれを母体に §3 の網羅で拡張。
- 各ケースに **ID** を付与し §7 トレーサビリティで受入条件に紐付ける。

## 6. PdM 受入基準・エビデンス・サインオフ
各フェーズ、PdM は**提示物（エビデンス）を見て定量で合否**。OK は数値とセットで記録。

| フェーズ | PdM への提示物（エビデンス） | 受入基準（PdMが確認する定量条件） | 合否 | サインオフ |

| **P2/2.5** | 操作/削除のロール別結果＋**案C連動の保存→再取得結果**＋削除の監査ログ | 付与のみ成功・未付与403=100%／連動が仕様一致／削除は確認＋付与のみ・監査100% | □OK □NG | |

- **質的条件（業務影響・現場確認）**：対象ロールの現場担当が「必要画面/データが見える」を確認しサイン（チェックリスト添付）。PdM はそのサインをエビデンスとする。
- **サインオフ条件**：当該フェーズの全行 OK ＋ §3 カバレッジ100% ＋ §7 トレーサビリティに未消化条件なし → 次フェーズ着手可。

## 8. 記録・運用
- 結果は「ケースID / OK・NG / 実値 / 日付 / 実行者」で記録（CIアーティファクト＋受入記録シート）。
- シャドー/越境/性能は**実測値で**判定（憶測で済ませない）。
- 機能の増減時は `feature_keys.json` を再生成し、§3 網羅と §7 トレーサビリティを更新。
